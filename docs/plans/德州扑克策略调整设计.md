你现在说的 `UTG FOLD - MP FOLD - CO FOLD - BTN` 这种字符串，适合当 **exact lookup key**，但不适合当系统的核心推理键。

- 当前行动者位置
- 上一个主动进攻者位置
- 当前面对的是：
  - unopened
  - limp

  - 3bet
  - 4bet
- 当前有几个 limper / caller
- hero 是否 IP / OOP

- `to_call`

也就是：  
**不要用“行动历史字符串”来代表局面，用“决策状态向量”来代表局面。**

## 二、偏离标准行动线时怎么办

### 1）动作偏离：用“动作族 + 最近邻状态”解决

我建议你把每一步动作先抽象成动作族：

- `FOLD`
- `LIMP`
- `CALL`
- `OPEN`
- `ISO`
- `3BET`
- `SQUEEZE`
- `4BET`
- `JAM`

然后做一个三层 fallback：

### 第一层：精确匹配
如果本地库里有 exact node，直接用。

### 第二层：最近邻节点
如果没有 exact node，就从你的本地 solver 节点里找 **同一类公共状态** 的最近邻节点，然后加权平均。

比如真实牌局：

- `UTG open 3bb`
- `MP call`
- 轮到 `BTN`

如果库里没有这个 exact line，就找这些近似节点：

- `EP open + 1 caller + BTN act`
- `UTG open + CO call + BTN act`
- `LJ open + HJ call + BTN act`

然后按距离加权：

$$
w_i = e^{-d(s, s_i)/\tau}
$$

$$
\pi_{base}(h,a) = \frac{\sum_i w_i \pi_i(h,a)}{\sum_i w_i}
$$

这里的距离 `d` 不要按字符串算，而要按这些特征算：

- aggressor position 是否接近
- caller 数量
- hero position
- IP/OOP
- facing family 是否相同
- size 是否接近
- remaining players 是否接近

这一步非常重要，因为你已经把 GTOWizard 全部下载到本地了，最有价值的不是“查表”，而是把这堆解当成 **样本库** 做最近邻泛化。

### 第三层：合成模板
如果连最近邻都不够，就用模板。

例如：

- `LIMP`：默认视为 **弱且 capped 的进入范围**
- `OPEN + COLD CALL`：默认视为
  - opener：相对 uncapped
  - caller：**condensed + capped**
- `多 limp`：默认多个弱进入范围，iso 倾向更 value-heavy

## 三、尺度偏离怎么办

尺度偏离要分成两个问题：

### 1）**价格效应**
这个一定要调。

比如标准 open 是 2bb，实际对手 open 到 3bb。  
即使对手范围不变，你的防守也必须更紧，因为 pot odds 变了。

所以你的映射最好不要只看 `size in bb`，而要看：

- `to_call / pot_after_call`
- `raise_to / previous_bet`
- `SPR` 的变化

### 2）**范围信号效应**
这个不能默认强行调。

很多玩家一直用 3bb open，只是个人习惯，不代表范围更强。  
所以：

- **默认先调价格，不调范围强弱**
- 只有当数据库证明这个玩家“size = range signal”时，再额外调范围

这是个很容易做错的点。  
很多程序会把“3bb open”自动当成“更强 range”，这在很多池子里是错的。

## 四、尺度映射的具体实现

### 1）如果 solver 有多个 size
直接做插值。

假设库里有 2bb 和 2.5bb，实际是 2.25bb：

$$
\pi(s) = \alpha \pi(2.0) + (1-\alpha)\pi(2.5)
$$

最好用 log-size 插值：

$$
\alpha = \frac{\ln(2.5)-\ln(2.25)}{\ln(2.5)-\ln(2.0)}
$$

### 2）如果实际 size 超出求解范围
例如库里只有 2bb，实际是 3bb：

- 先取最近 size 的 solver 策略做基准
- 再加一个 **价格修正**

一个很好用的经验是：

- **大 size**：先砍掉最边缘 continue
- **小 size**：先加回最边缘 continue

而且不同手牌家族，弹性不同：

- 最先被大 size 打掉的：弱 offsuit broadway、弱 ATo/KJo/QTo 这类易被支配牌
- 相对更能保留的：小中 pocket、suited broadway、suited connector
- 最不受影响的：顶端 value

也就是说，尺度偏离不是“整体平移”，而是“边界先动”。

## 五、没有 limp 线怎么办

这个是你系统里必须单独做的 family，不能硬映射成 fold/open。

### 我建议把 limp 当成一个独立节点家族：

你需要估计 limper 的三个后续倾向：

- `limp-fold`
- `limp-call`
- `limp-reraise`

然后 hero 的动作不是简单 `open/fold`，而是：

- `iso`
- `overlimp`
- `fold`

### 默认逻辑：

#### 对手 limp-fold 高
你要 **iso 更宽**

因为你赚的是：

- 立即弃牌率
- 位置优势
- 额外 dead money

#### 对手 limp-call 高，但 limp-reraise 低
你要 **iso 更线性、更偏价值**

扩大：

- broadway
- suited ace
- medium pair
- suited king / suited queen 中较好部分

减少：

- 纯垃圾 offsuit

#### 对手 limp-reraise 高
你要减少最弱 iso，增加 overlimp

### 合成 limp range 的默认形状
在没有足够 showdowns 之前，默认 limp range 应该是：

- 弱 Ax / 弱 broadway
- 小中 pocket
- suited connector / suited gapper
- 一部分中等牌力
- 极少量 trap

也就是说：

> **默认 limp range 是弱且 capped，但不能设成完全无强牌。**

## 六、没有 “open + cold call” 线怎么办

这个也不要硬套成 “只面对 open”。

例如：

- `UTG open`
- `MP call`
- 你在 `BTN`

这里 caller 的范围通常不是“最强的跟注牌”，而是：

- 顶端强牌很多已经 3bet 了
- 底端垃圾很多已经 fold 了
- 剩下的是一段 **中间密集区间**

也就是你说的那句，本质上完全正确：

> **call range 不是 top-x%，而是中间收缩、两头变薄的 condensed range。**

### 对 hero 的调整方向：

#### 1）减少被支配、怕多人的平跟
比如一些边缘 offsuit broadway 会掉得很快。

#### 2）保留/增加适合多人的手
比如：

- 小中 pocket
- suited connector
- suited broadway

前提是你有位置、身后 squeeze 风险不高、有效筹码允许。

#### 3）增加 squeeze，但不是瞎加
要看两件事：

- opener 会不会对 squeeze 过度弃牌
- caller 会不会对 squeeze 过度弃牌

如果两个人都容易 fold，Axs/Kxs 这类 blocker bluff squeeze 的权重可以提高。  
如果 caller 是 loose-passive 且不爱 fold，那 squeeze 更应该偏 value / linear。

## 七、怎么根据玩家倾向调整策略

这里不要直接把玩家 stat 写死成“比标准紧 4%，那我所有继续范围减 4%”。  
更好的方式是：

## 1）先用历史数据得到“玩家版动作频率”

对每个玩家、每个 node family，维护：

- open%
- limp%
- call%
- 3bet%
- 4bet%
- fold to 3bet%
- fold to steal%
- squeeze%
- limp-fold%
- limp-reraise%

而且要做平滑，别让小样本乱飞。

### 推荐平滑方式
$$
\hat p = \frac{n_{act} + k \mu}{N + k}
$$

- `μ`：群体均值 / 同类玩家均值
- `k`：先验强度
- `N`：这个玩家在这个节点的样本数

这样小样本先别过度 exploit，大样本再放开。

---

## 八、最关键的一步：用“概率倾斜”而不是手工删牌

你已经有 solver 基准策略 `π_gto(h,a)`。

现在某个玩家在 `UTG unopened` 的 open 频率只有 10%，你要把 solver 的 UTG open 策略倾斜成“这个玩家版”的 open 策略。

### 二元动作（open / fold）可以这样做：

$$
p_{new}(h,\text{open}) = \sigma(\text{logit}(p_{gto}(h,\text{open})) + \lambda)
$$

通过调一个标量 `λ`，让总体 open 频率变成 10%。

这个方法特别好，因为它会自然地：

- 紧的时候先删掉边界 open
- 松的时候先加回边界 open
- 保留 solver 原来的手牌排序

### 多动作（fold / call / 3bet）可以这样做：

$$
p_{new}(h,a)=\frac{\exp(\log p_{gto}(h,a)+\lambda_a)}{\sum_b \exp(\log p_{gto}(h,b)+\lambda_b)}
$$

然后求一组 `λ_a`，使得整体动作频率匹配玩家历史数据。

---

## 九、为什么这个方法特别适合“估计对手范围”

因为你不需要“猜这 10% 是哪 10% 手牌”，你是直接从 solver 的动作结构出发做投影。

### 范围更新公式

假设某玩家当前节点前的先验范围是 `R_before(h)`，  
观察到他做了动作 `a*`，  
那更新后的范围就是：

$$
R_{after}(h) \propto R_{before}(h)\cdot p_{new}(h,a^*)
$$

这就是标准贝叶斯更新。

### 好处
- open range 会自然偏“线性强牌”
- call range 会自然偏“中间密集”
- 3bet range 会自然带 blocker / value 结构
- 不需要你手写一大堆“这手该不该在 call range 里”

不是手写，而是：

1. 取你本地 solver 的 UTG unopened 基准 open 概率 `p_gto(h, open)`
2. 用上面的 logit shift，把总体频率校准到 10%
3. 得到玩家版的 `p_new(h, open)`
4. 观察到他真的 open 了，则：
   $$
   R(h|open) \propto p_{new}(h,open)
   $$

## 十二、你的 exploit 层应该怎么做

不要直接覆盖 GTO，应该做“有置信度的偏移”。

### 形式上可以这样写：

$$
\text{logit}\, \pi_{hero}(h,a)
=
\text{logit}\, \pi_{gto}(h,a)
+
c \cdot \Delta_{exploit}(h,a)
$$

其中：

- `c`：置信度，样本越多越大
- `Δ_exploit`：根据对手倾向算出来的 exploit 偏移

### 典型规则

#### 1）UTG 比标准紧很多
- 对他的 open：**收紧 flat**
- **收紧 bluff 3bet**
- value 3bet 更纯
- 如果他 `fold to 3bet` 仍然很高，再把一部分 blocker bluff 加回来

也就是：
**“范围强” 和 “会不会过度弃牌” 要分开看。**

#### 2）UTG 比标准松很多
- 你的 defend 可以变宽
- 你的 value 3bet 可以变宽
- 如果他 open wide 且 fold to 3bet 高，bluff 3bet 也可以放宽

#### 3）BTN 偷盲时，SB/BB 防守不足
- BTN open 变宽
- 最先加进来的，是原来 open/fold 边界处的偷盲牌
- 如果他们只是 **call 少**，但 **3bet 很多**，那就不能无脑放太宽，要改成更能抗 3bet 的结构

#### 4）SB/BB 过度平跟、很少 3bet
- 仍然可以宽偷
- 但应更偏向：
  - suited hand
  - broadway
  - 有 postflop 可实现率的牌
- 少放纯垃圾 offsuit

#### 5）对手冷跟过多
- squeeze 频率可以提高
- 但更偏线性/value
- 尺度也应该更大

## 3. 离线预处理要先做好

这是最值钱的一步。先别急着写 runtime。

### 3.1 把所有 GTOWizard 节点标准化

每个节点都提取成统一特征：

- 当前行动者位置
- 决策家族 `family`
- aggressor 位置
- limper 数
- caller 数
- IP/OOP
- 有效筹码 bucket
- `raise_to_bb`
- `risk_over_pot`
- `players_left`

然后做索引：

```python
index_key = (family, actor_pos, effective_bucket)
```

这样查最近邻时，不会在全库乱找。

---

### 3.2 预计算“动作专属排序分数”

这一点非常关键。

你后面要估算“UTG open 10% 是什么范围”“MP cold call 7% 是什么范围”，不能按全局牌力排，要按 **动作专属分数** 排。

离线对每个 archetype 统计：

```python
score_open[h]  = 平均 solver 中该手在这个 archetype 的 OPEN 概率
score_call[h]  = 平均 solver 中该手在这个 archetype 的 CALL 概率
score_3bet[h]  = 平均 solver 中该手在这个 archetype 的 3BET 概率
score_4bet[h]  = 平均 solver 中该手在这个 archetype 的 4BET 概率
```

例如：

- `score_open` 会偏线性强牌
- `score_call` 会天然更 condensed
- `score_3bet` 会混入 blocker/value 结构

所以后面即使没有 exact node，你也能靠这个分数构造一个像样的范围。

---

### 3.3 预计算 size 弹性

如果你的库里同一类节点有多个 size，离线拟合“继续频率对 size 的弹性”。

例如 BTN 面对 open：

$$
\eta_{arch} =
\frac{continue(s_2)-continue(s_1)}{\log(s_2/s_1)}
$$

运行时实际从 2bb 偏到 3bb 时，不用硬猜，直接按这个斜率外推。

## 5. NodeResolver 怎么写

### 5.1 精确匹配

先尝试 exact：

```python
node = repo.get_exact(canonical_key_from_state(state))
if node:
    return node
```

### 5.2 最近邻匹配

找不到 exact，就在同 family、同 actor_pos、同 eff_bucket 下找最近邻。

距离函数建议这样写：

```python
def state_distance(q, n):
    d = 0.0
    d += 4.0 * (q.family != n.family)
    d += 2.0 * pos_distance(q.actor_pos, n.actor_pos)
    d += 2.0 * pos_distance(q.aggressor_pos, n.aggressor_pos)
    d += 1.5 * abs(q.caller_count - n.caller_count)
    d += 1.5 * abs(q.limper_count - n.limper_count)
    d += 1.0 * abs(q.players_left - n.players_left)
    d += 1.0 * abs(float(q.ip) - float(n.ip))
    d += 1.0 * abs(np.log((q.risk_over_pot or 1.0) / (n.risk_over_pot or 1.0)))
    d += 1.0 * abs(np.log(((q.raise_size_bb or 2.0) / (n.raise_size_bb or 2.0))))
    return d
```

取前 `k=3~5` 个节点，做加权平均：

$$
w_i = e^{-d_i / \tau}
$$

$$
P_{base}(h,a) = \frac{\sum_i w_i P_i(h,a)}{\sum_i w_i}
$$

## 6. 模板策略怎么做

## 6.1 limp 模板

如果 solver 库没有 limp，你必须自己定义 limp 模板家族。

先给 limp 一个默认排序分数 `score_limp`，第一版可以这样做：

- 小中 pocket：高
- suited connector / gapper：高
- 弱 suited ace：中高
- 弱 broadway：中
- premium：很低，但留少量 trap
- 纯垃圾 offsuit：低

运行时如果某玩家在某位置 limp 率是 `x%`，就按 `score_limp` 构造他的 limp 范围：

```python
def build_template_range(score, target_freq):
    order = np.argsort(-score)
    w = np.zeros_like(score)
    cum = 0.0
    for i in order:
        add = hand_weight(i)   # 169/1326 对应的组合权重
        if cum + add <= target_freq:
            w[i] = 1.0
            cum += add
        else:
            w[i] = max(0.0, (target_freq - cum) / add)
            break
    w /= w.sum()
    return w
```

hero 面对 limp 时：

- 对手 `limp-fold` 高：iso 更宽
- 对手 `limp-call` 高：iso 更线性、更偏价值
- 对手 `limp-reraise` 高：减弱最边缘 iso，增加 overlimp/fold

---

## 6.2 open + cold call 模板

这个模板很重要。

对 caller 的范围，不要用“top x%”，而要用 `score_call` 去构造：

```python
caller_range = build_template_range(score_call[arch], target_call_freq)
```

这样 caller 的范围自然会是：

- 顶端少一截，因为很多顶端已 3bet
- 底端少一截，因为弱牌已 fold
- 中间密集

这正是你要的 condensed 结构。

## 7. 根据玩家倾向调整 base policy

这一步不要“手工删牌”，而要做 **动作概率校准**。

### 7.1 先得到玩家目标动作频率

对某个上下文 `ctx`，先算 solver 基准动作频率 `mu_solver(a|ctx)`。

再从玩家历史中做平滑估计：

$$
\hat p_{player}(a|ctx) = \frac{n_a + k \mu_{pool}(a|ctx)}{N + k}
$$

然后用样本置信度跟 solver 基准混合：

$$
c = \frac{N}{N + K}
$$

$$
t(a|ctx) = (1-c)\mu_{solver}(a|ctx) + c \hat p_{player}(a|ctx)
$$

这个 `t(a|ctx)` 才是运行时真正要匹配的目标动作频率。

---

### 7.2 二元动作校准

比如 unopened 只看 `OPEN/FOLD`：

$$
q(h,\text{OPEN}) =
\sigma(\logit(p_{base}(h,\text{OPEN})) + \lambda)
$$

通过二分搜索 `\lambda`，让整体 open 率等于目标：

$$
\sum_h R(h)\, q(h,\text{OPEN}) = t(\text{OPEN})
$$

伪代码：

```python
def calibrate_binary(base_open, prior_range, target_open):
    eps = 1e-6
    lo, hi = -8.0, 8.0
    x = np.clip(base_open, eps, 1 - eps)

    for _ in range(40):
        mid = (lo + hi) / 2
        q = 1 / (1 + np.exp(-(np.log(x / (1 - x)) + mid)))
        freq = float((prior_range * q).sum())
        if freq > target_open:
            hi = mid
        else:
            lo = mid
    return q
```

---

### 7.3 多动作校准

比如 `FOLD/CALL/3BET`：

$$
q(h,a)=\frac{\exp(\log p_{base}(h,a)+\lambda_a)}
{\sum_b \exp(\log p_{base}(h,b)+\lambda_b)}
$$

求一组 `\lambda_a`，让整体动作频率匹配 `t(a)`。

```python
def calibrate_multiaction(base_probs, prior_range, target, iters=80, lr=0.5):
    eps = 1e-8
    A = base_probs.shape[1]
    lam = np.zeros(A)

    for _ in range(iters):
        logits = np.log(np.clip(base_probs, eps, 1.0)) + lam[None, :]
        logits -= logits.max(axis=1, keepdims=True)
        q = np.exp(logits)
        q /= q.sum(axis=1, keepdims=True)

        marg = prior_range @ q
        grad = target - marg
        lam += lr * grad

        if np.max(np.abs(grad)) < 1e-4:
            break

    return q
```

这个方法的好处是：  
紧一点时，会先砍掉边界 combo；松一点时，会先加回边界 combo；不会破坏 solver 原始排序。

## 8. 范围更新怎么写

每观察到一个动作，就递推更新一次。

$$
R_{after}(h) \propto R_{before}(h)\cdot q(h,a^*)
$$

伪代码：

```python
def bayes_update(prior_range, action_probs_for_combo, observed_action_idx):
    post = prior_range * action_probs_for_combo[:, observed_action_idx]
    s = post.sum()
    if s <= 1e-12:
        return prior_range.copy()
    return post / s
```

整手牌开局到 hero 决策前，把所有已行动对手都更新一遍。

## 9. 非标准尺度怎么处理

尺度偏离分两部分：

## 9.1 价格效应

这个必须调。

例如标准 open 2bb，实际 3bb。  
先映射到最近 size 的 solver node，再按 size 弹性修正边界继续频率。

$$
continue_{target}
=
continue_{ref} + \eta_{arch}\cdot \log(size_{act}/size_{ref})
$$

如果 `size_act > size_ref`，通常 `continue_target` 会下降。

然后不是整体缩小，而是按 `score_call` / `score_defend` 从边界开始删。

---

## 9.2 范围信号效应

这个不能默认瞎调。

第一版建议：

- **默认只调价格，不调强弱**
- 只有当你有足够证据时，才把 size 当范围信号

证据可以来自：

- 该玩家在同位置的 size 选择非常稳定
- 或你有 showdown 数据证明“大 size 更强”

### 一个更好的做法

如果你的本地 solver 节点本来就有多个 size 动作，比如：

- `OPEN_2.0`
- `OPEN_2.5`
- `OPEN_3.0`

那观察到对手用了某个 size，本身就能更新范围：

$$
R(h|OPEN\_3.0) \propto R(h)\cdot p(h, OPEN\_3.0)
$$

如果实际是 2.7bb，没有 exact size，就对 size action 插值。

这比“3bb = 更强”这种粗暴规则好很多。

## 10. exploit 层怎么写

hero 也不要直接手工删牌，还是对 base policy 做偏移。

$$
\logit \pi_{hero}(h,a)
=
\logit \pi_{base}(h,a)
+
c \cdot \Delta(h,a)
$$

其中 `c` 是对手样本置信度，`Δ(h,a)` 来自 exploit 特征。

### 10.1 先预计算手牌特征

每手牌准备一些分数：

- `playability_score`
- `domination_risk`
- `blocker_score`
- `multiway_score`
- `linear_value_score`

这些第一版可以手工规则，后面再学。

---

### 10.2 常见 exploit 规则

#### 对紧 UTG open

- 降低 flat，特别是边缘 offsuit broadway
- 降低 bluff 3bet
- 保留最强 value 3bet
- 若对方 `fold_to_3bet` 很高，再给 blocker bluff 一点权重

示意：

```python
delta_call  = - conf * opener_strength_excess * domination_risk
delta_3bet  = - conf * opener_strength_excess * bluff_score
delta_3bet += + conf * overfold_to_3bet * blocker_score
```

#### 对松 UTG open

- 放宽 defend
- 放宽 value 3bet
- 若对方 open wide 且对 3bet 弃牌多，增加 blocker bluff 3bet

#### BTN 偷盲，SB/BB 防守不足

- 提高 open 频率
- 优先加入有 blocker、可实现率好的偷盲牌
- 少加纯垃圾 offsuit

```python
delta_open = conf * blind_underdefense * (
    0.5 * blocker_score + 0.5 * playability_score - 0.4 * domination_risk
)
```

#### 对手 cold call 过多

- squeeze 可以提高
- 但更偏 linear / value
- size 也应该更大

## 11. 你最关心的：怎么从“UTG open 10%”估计范围

最好的实现不是手写一堆牌，而是：

### 方法 A：有 solver 先验时

1. 取 `UTG unopened` 的 base open 概率 `p_base(h, OPEN)`
2. 用上面的 binary calibration，把整体 open 率压到 10%
3. 观察到他真的 open 了，再做一次 Bayes 更新：

$$
R(h|open) \propto R(h)\cdot q(h, open)
$$

这就是你程序里最该用的方法。

---

### 方法 B：没有 exact node 时

用离线预计算的 `score_open[UTG, unopened]` 构造 10% 模板范围：

```python
utg_open_10 = build_template_range(score_open_utg, 0.10)
```

这个模板自然比“全局 top10%”更像 solver。

---

### 为什么 call 不能按 top-x%

因为 call 不是线性顶端范围。

正确做法是：

```python
mp_call_7 = build_template_range(score_call_mp_vs_utg, 0.07)
```

这样构造出的 call range 会天然是中等牌力密集区，而不是把 `AA/KK/AK` 塞满。

## 12. 推荐的上下文分层回退

玩家 stats 很容易小样本，所以每个查询都做 backoff。

例如你要查：

`BTN facing CO open unopened, 100bb, IP`

按 4 层取：

1. `exact`: BTN vs CO open, 100bb
2. `mid`: BTN vs LP open
3. `coarse`: BTN vs open
4. `global`: 所有玩家 BTN facing open

然后按样本量加权：

$$
\hat p = \frac{\sum_l c_l \hat p_l}{\sum_l c_l},
\quad
c_l = \frac{N_l}{N_l + K_l}
$$

这样小样本不会乱跳。

## 15. 最后给你一个完整伪代码骨架

```python
def get_hero_decision(hero_hand, history, player_profiles, stacks):
    state = build_runtime_state(history, stacks)

    # 1) 基准策略
    base_policy = node_resolver.resolve(state)

    # 2) 更新所有已行动对手范围
    opp_ranges = {}
    for player in acted_players_before_hero(history):
        r = initial_range()
        for event, ctx in player_action_sequence(player, history):
            target = player_model.get_target_mix(player, ctx, base_policy)
            combo_action_probs = calibrate_policy_for_player(
                base_policy_for_ctx(ctx),
                prior_range=r,
                target_mix=target,
                observed_size=event.size_bb
            )
            a_idx = action_index(combo_action_probs.actions, event)
            r = bayes_update(r, combo_action_probs.matrix, a_idx)
        opp_ranges[player.player_id] = r

    # 3) 调整 hero 策略
    hero_policy = exploit_adjuster.adjust(
        base_policy=base_policy,
        state=state,
        hero_hand=hero_hand,
        opp_ranges=opp_ranges,
        player_profiles=player_profiles
    )

    return hero_policy.action_probs_for(hero_hand)
```

---

这套设计的核心优点是统一：

- **非标准行动线**：靠 `state + resolver`
- **非标准尺度**：靠 `size interpolation + elasticity`
- **玩家倾向调整**：靠 `target mix + calibration`
- **对手范围估计**：靠 `Bayes update`
- **call range 不是 top-x%**：靠 `score_call` 模板

下一步最合适的是先把 `RuntimeState / NodeResolver / PlayerModel / RangeEngine` 这 4 个类的 Python 骨架定下来，再接你的本地 GTOWizard 数据。
