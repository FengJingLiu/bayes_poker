也就是：

## 二、偏离标准行动线时怎么办

### 1）动作偏离：用“动作族 + 最近邻状态”解决

我建议你把每一步动作先抽象成动作族：

- `FOLD`
- `LIMP`
- `CALL`
- `OPEN`
- `ISO`
- `3BET`
- `SQUEEZE`
- `4BET`
- `JAM`

然后做一个三层 fallback：

### 第一层：精确匹配
如果本地库里有 exact node，直接用。

### 第二层：最近邻节点
如果没有 exact node，就从你的本地 solver 节点里找 **同一类公共状态** 的最近邻节点，然后加权平均。

比如真实牌局：

- `UTG open 3bb`
- `MP call`
- 轮到 `BTN`

如果库里没有这个 exact line，就找这些近似节点：

- `EP open + 1 caller + BTN act`
- `UTG open + CO call + BTN act`
- `LJ open + HJ call + BTN act`

然后按距离加权：

$$
w_i = e^{-d(s, s_i)/\tau}
$$

$$
\pi_{base}(h,a) = \frac{\sum_i w_i \pi_i(h,a)}{\sum_i w_i}
$$

这里的距离 `d` 不要按字符串算，而要按这些特征算：

- aggressor position 是否接近
- caller 数量
- hero position
- IP/OOP
- facing family 是否相同
- size 是否接近
- remaining players 是否接近

这一步非常重要，因为你已经把 GTOWizard 全部下载到本地了，最有价值的不是“查表”，而是把这堆解当成 **样本库** 做最近邻泛化。

### 第三层：合成模板
如果连最近邻都不够，就用模板。

例如：

- `LIMP`：默认视为 **弱且 capped 的进入范围**
- `OPEN + COLD CALL`：默认视为
  - opener：相对 uncapped
  - caller：**condensed + capped**
- `多 limp`：默认多个弱进入范围，iso 倾向更 value-heavy

## 三、尺度偏离怎么办

尺度偏离要分成两个问题：

### 1）**价格效应**
这个一定要调。

比如标准 open 是 2bb，实际对手 open 到 3bb。  
即使对手范围不变，你的防守也必须更紧，因为 pot odds 变了。

所以你的映射最好不要只看 `size in bb`，而要看：

- `to_call / pot_after_call`
- `raise_to / previous_bet`
- `SPR` 的变化

### 2）**范围信号效应**
这个不能默认强行调。

很多玩家一直用 3bb open，只是个人习惯，不代表范围更强。  
所以：

- **默认先调价格，不调范围强弱**
- 只有当数据库证明这个玩家“size = range signal”时，再额外调范围

这是个很容易做错的点。  
很多程序会把“3bb open”自动当成“更强 range”，这在很多池子里是错的。

## 四、尺度映射的具体实现

### 1）如果 solver 有多个 size
直接做插值。

假设库里有 2bb 和 2.5bb，实际是 2.25bb：

$$
\pi(s) = \alpha \pi(2.0) + (1-\alpha)\pi(2.5)
$$

最好用 log-size 插值：

$$
\alpha = \frac{\ln(2.5)-\ln(2.25)}{\ln(2.5)-\ln(2.0)}
$$

### 2）如果实际 size 超出求解范围
例如库里只有 2bb，实际是 3bb：

- 先取最近 size 的 solver 策略做基准
- 再加一个 **价格修正**

一个很好用的经验是：

- **大 size**：先砍掉最边缘 continue
- **小 size**：先加回最边缘 continue

而且不同手牌家族，弹性不同：

- 最先被大 size 打掉的：弱 offsuit broadway、弱 ATo/KJo/QTo 这类易被支配牌
- 相对更能保留的：小中 pocket、suited broadway、suited connector
- 最不受影响的：顶端 value

也就是说，尺度偏离不是“整体平移”，而是“边界先动”。

## 五、没有 limp 线怎么办

这个是你系统里必须单独做的 family，不能硬映射成 fold/open。

### 我建议把 limp 当成一个独立节点家族：

你需要估计 limper 的三个后续倾向：

- `limp-fold`
- `limp-call`
- `limp-reraise`

然后 hero 的动作不是简单 `open/fold`，而是：

- `iso`
- `overlimp`
- `fold`

### 默认逻辑：

#### 对手 limp-fold 高
你要 **iso 更宽**

因为你赚的是：

- 立即弃牌率
- 位置优势
- 额外 dead money

#### 对手 limp-call 高，但 limp-reraise 低
你要 **iso 更线性、更偏价值**

扩大：

- broadway
- suited ace
- medium pair
- suited king / suited queen 中较好部分

减少：

- 纯垃圾 offsuit

#### 对手 limp-reraise 高
你要减少最弱 iso，增加 overlimp

### 合成 limp range 的默认形状
在没有足够 showdowns 之前，默认 limp range 应该是：

- 弱 Ax / 弱 broadway
- 小中 pocket
- suited connector / suited gapper
- 一部分中等牌力
- 极少量 trap

也就是说：

> **默认 limp range 是弱且 capped，但不能设成完全无强牌。**

## 六、没有 “open + cold call” 线怎么办

这个也不要硬套成 “只面对 open”。

例如：

- `UTG open`
- `MP call`
- 你在 `BTN`

这里 caller 的范围通常不是“最强的跟注牌”，而是：

- 顶端强牌很多已经 3bet 了
- 底端垃圾很多已经 fold 了
- 剩下的是一段 **中间密集区间**

也就是你说的那句，本质上完全正确：

> **call range 不是 top-x%，而是中间收缩、两头变薄的 condensed range。**

### 对 hero 的调整方向：

#### 1）减少被支配、怕多人的平跟
比如一些边缘 offsuit broadway 会掉得很快。

#### 2）保留/增加适合多人的手
比如：

- 小中 pocket
- suited connector
- suited broadway

前提是你有位置、身后 squeeze 风险不高、有效筹码允许。

#### 3）增加 squeeze，但不是瞎加
要看两件事：

- opener 会不会对 squeeze 过度弃牌
- caller 会不会对 squeeze 过度弃牌

如果两个人都容易 fold，Axs/Kxs 这类 blocker bluff squeeze 的权重可以提高。  
如果 caller 是 loose-passive 且不爱 fold，那 squeeze 更应该偏 value / linear。

## 七、怎么根据玩家倾向调整策略

这里不要直接把玩家 stat 写死成“比标准紧 4%，那我所有继续范围减 4%”。  
更好的方式是：

## 1）先用历史数据得到“玩家版动作频率”

对每个玩家、每个 node family，维护：

- open%
- limp%
- call%
- 3bet%
- 4bet%
- fold to 3bet%
- fold to steal%
- squeeze%
- limp-fold%
- limp-reraise%

而且要做平滑，别让小样本乱飞。

### 推荐平滑方式
$$
\hat p = \frac{n_{act} + k \mu}{N + k}
$$

- `μ`：群体均值 / 同类玩家均值
- `k`：先验强度
- `N`：这个玩家在这个节点的样本数

这样小样本先别过度 exploit，大样本再放开。

---

## 八、最关键的一步：用“概率倾斜”而不是手工删牌

你已经有 solver 基准策略 `π_gto(h,a)`。

现在某个玩家在 `UTG unopened` 的 open 频率只有 10%，你要把 solver 的 UTG open 策略倾斜成“这个玩家版”的 open 策略。

### 二元动作（open / fold）可以这样做：

$$
p_{new}(h,\text{open}) = \sigma(\text{logit}(p_{gto}(h,\text{open})) + \lambda)
$$

通过调一个标量 `λ`，让总体 open 频率变成 10%。

这个方法特别好，因为它会自然地：

- 紧的时候先删掉边界 open
- 松的时候先加回边界 open
- 保留 solver 原来的手牌排序

### 多动作（fold / call / 3bet）可以这样做：

$$
p_{new}(h,a)=\frac{\exp(\log p_{gto}(h,a)+\lambda_a)}{\sum_b \exp(\log p_{gto}(h,b)+\lambda_b)}
$$

然后求一组 `λ_a`，使得整体动作频率匹配玩家历史数据。

---

## 九、为什么这个方法特别适合“估计对手范围”

因为你不需要“猜这 10% 是哪 10% 手牌”，你是直接从 solver 的动作结构出发做投影。

### 范围更新公式

假设某玩家当前节点前的先验范围是 `R_before(h)`，  
观察到他做了动作 `a*`，  
那更新后的范围就是：

$$
R_{after}(h) \propto R_{before}(h)\cdot p_{new}(h,a^*)
$$

这就是标准贝叶斯更新。

### 好处
- open range 会自然偏“线性强牌”
- call range 会自然偏“中间密集”
- 3bet range 会自然带 blocker / value 结构
- 不需要你手写一大堆“这手该不该在 call range 里”

不是手写，而是：

1. 取你本地 solver 的 UTG unopened 基准 open 概率 `p_gto(h, open)`
2. 用上面的 logit shift，把总体频率校准到 10%
3. 得到玩家版的 `p_new(h, open)`
4. 观察到他真的 open 了，则：
   $$
   R(h|open) \propto p_{new}(h,open)
   $$

## 十二、你的 exploit 层应该怎么做

不要直接覆盖 GTO，应该做“有置信度的偏移”。

### 形式上可以这样写：

$$
\text{logit}\, \pi_{hero}(h,a)
=
\text{logit}\, \pi_{gto}(h,a)
+
c \cdot \Delta_{exploit}(h,a)
$$

其中：

- `c`：置信度，样本越多越大
- `Δ_exploit`：根据对手倾向算出来的 exploit 偏移

### 典型规则

#### 1）UTG 比标准紧很多
- 对他的 open：**收紧 flat**
- **收紧 bluff 3bet**
- value 3bet 更纯
- 如果他 `fold to 3bet` 仍然很高，再把一部分 blocker bluff 加回来

也就是：
**“范围强” 和 “会不会过度弃牌” 要分开看。**

#### 2）UTG 比标准松很多
- 你的 defend 可以变宽
- 你的 value 3bet 可以变宽
- 如果他 open wide 且 fold to 3bet 高，bluff 3bet 也可以放宽

#### 3）BTN 偷盲时，SB/BB 防守不足
- BTN open 变宽
- 最先加进来的，是原来 open/fold 边界处的偷盲牌
- 如果他们只是 **call 少**，但 **3bet 很多**，那就不能无脑放太宽，要改成更能抗 3bet 的结构

#### 4）SB/BB 过度平跟、很少 3bet
- 仍然可以宽偷
- 但应更偏向：
  - suited hand
  - broadway
  - 有 postflop 可实现率的牌
- 少放纯垃圾 offsuit

#### 5）对手冷跟过多
- squeeze 频率可以提高
- 但更偏线性/value
- 尺度也应该更大

## 3. 离线预处理要先做好

这是最值钱的一步。先别急着写 runtime。

### 3.1 把所有 GTOWizard 节点标准化

每个节点都提取成统一特征：

- 当前行动者位置
- 决策家族 `family`
- aggressor 位置
- limper 数
- caller 数
- IP/OOP
- 有效筹码 bucket
- `raise_to_bb`
- `risk_over_pot`
- `players_left`

然后做索引：

```python
index_key = (family, actor_pos, effective_bucket)
```

这样查最近邻时，不会在全库乱找。

---

### 3.2 预计算“动作专属排序分数”

这一点非常关键。

你后面要估算“UTG open 10% 是什么范围”“MP cold call 7% 是什么范围”，不能按全局牌力排，要按 **动作专属分数** 排。

离线对每个 archetype 统计：

```python
score_open[h]  = 平均 solver 中该手在这个 archetype 的 OPEN 概率
score_call[h]  = 平均 solver 中该手在这个 archetype 的 CALL 概率
score_3bet[h]  = 平均 solver 中该手在这个 archetype 的 3BET 概率
score_4bet[h]  = 平均 solver 中该手在这个 archetype 的 4BET 概率
```

例如：

- `score_open` 会偏线性强牌
- `score_call` 会天然更 condensed
- `score_3bet` 会混入 blocker/value 结构

所以后面即使没有 exact node，你也能靠这个分数构造一个像样的范围。

---

### 3.3 预计算 size 弹性

如果你的库里同一类节点有多个 size，离线拟合“继续频率对 size 的弹性”。

例如 BTN 面对 open：

$$
\eta_{arch} =
\frac{continue(s_2)-continue(s_1)}{\log(s_2/s_1)}
$$

运行时实际从 2bb 偏到 3bb 时，不用硬猜，直接按这个斜率外推。

## 5. NodeResolver 怎么写

### 5.1 精确匹配

### 5.2 最近邻匹配

找不到 exact，就在同 family、同 actor_pos、同 eff_bucket 下找最近邻。

距离函数建议这样写：

```python
def state_distance(q, n):
    d = 0.0
    d += 4.0 * (q.family != n.family)
    d += 2.0 * pos_distance(q.actor_pos, n.actor_pos)
    d += 2.0 * pos_distance(q.aggressor_pos, n.aggressor_pos)
    d += 1.5 * abs(q.caller_count - n.caller_count)
    d += 1.5 * abs(q.limper_count - n.limper_count)
    d += 1.0 * abs(q.players_left - n.players_left)
    d += 1.0 * abs(float(q.ip) - float(n.ip))
    d += 1.0 * abs(np.log((q.risk_over_pot or 1.0) / (n.risk_over_pot or 1.0)))
    d += 1.0 * abs(np.log(((q.raise_size_bb or 2.0) / (n.raise_size_bb or 2.0))))
    return d
```

取前 `k=3~5` 个节点，做加权平均：

$$
w_i = e^{-d_i / \tau}
$$

$$
P_{base}(h,a) = \frac{\sum_i w_i P_i(h,a)}{\sum_i w_i}
$$

## 6. 模板策略怎么做

## 6.1 limp 模板

如果 solver 库没有 limp，你必须自己定义 limp 模板家族。

先给 limp 一个默认排序分数 `score_limp`，第一版可以这样做：

- 小中 pocket：高
- suited connector / gapper：高
- 弱 suited ace：中高
- 弱 broadway：中
- premium：很低，但留少量 trap
- 纯垃圾 offsuit：低

运行时如果某玩家在某位置 limp 率是 `x%`，就按 `score_limp` 构造他的 limp 范围：

```python
def build_template_range(score, target_freq):
    order = np.argsort(-score)
    w = np.zeros_like(score)
    cum = 0.0
    for i in order:
        add = hand_weight(i)   # 169/1326 对应的组合权重
        if cum + add <= target_freq:
            w[i] = 1.0
            cum += add
        else:
            w[i] = max(0.0, (target_freq - cum) / add)
            break
    w /= w.sum()
    return w
```

hero 面对 limp 时：

- 对手 `limp-fold` 高：iso 更宽
- 对手 `limp-call` 高：iso 更线性、更偏价值
- 对手 `limp-reraise` 高：减弱最边缘 iso，增加 overlimp/fold

---

## 6.2 open + cold call 模板

这个模板很重要。

对 caller 的范围，不要用“top x%”，而要用 `score_call` 去构造：

```python
caller_range = build_template_range(score_call[arch], target_call_freq)
```

这样 caller 的范围自然会是：

- 顶端少一截，因为很多顶端已 3bet
- 底端少一截，因为弱牌已 fold
- 中间密集

这正是你要的 condensed 结构。

## 7. 根据玩家倾向调整 base policy

这一步不要“手工删牌”，而要做 **动作概率校准**。

### 7.1 先得到玩家目标动作频率

对某个上下文 `ctx`，先算 solver 基准动作频率 `mu_solver(a|ctx)`。

再从玩家历史中做平滑估计：

$$
\hat p_{player}(a|ctx) = \frac{n_a + k \mu_{pool}(a|ctx)}{N + k}
$$

然后用样本置信度跟 solver 基准混合：

$$
c = \frac{N}{N + K}
$$

$$
t(a|ctx) = (1-c)\mu_{solver}(a|ctx) + c \hat p_{player}(a|ctx)
$$

这个 `t(a|ctx)` 才是运行时真正要匹配的目标动作频率。

---

### 7.2 二元动作校准

比如 unopened 只看 `OPEN/FOLD`：

$$
q(h,\text{OPEN}) =
\sigma(\logit(p_{base}(h,\text{OPEN})) + \lambda)
$$

通过二分搜索 `\lambda`，让整体 open 率等于目标：

$$
\sum_h R(h)\, q(h,\text{OPEN}) = t(\text{OPEN})
$$

伪代码：

```python
def calibrate_binary(base_open, prior_range, target_open):
    eps = 1e-6
    lo, hi = -8.0, 8.0
    x = np.clip(base_open, eps, 1 - eps)

    for _ in range(40):
        mid = (lo + hi) / 2
        q = 1 / (1 + np.exp(-(np.log(x / (1 - x)) + mid)))
        freq = float((prior_range * q).sum())
        if freq > target_open:
            hi = mid
        else:
            lo = mid
    return q
```

---

### 7.3 多动作校准

比如 `FOLD/CALL/3BET`：

$$
q(h,a)=\frac{\exp(\log p_{base}(h,a)+\lambda_a)}
{\sum_b \exp(\log p_{base}(h,b)+\lambda_b)}
$$

求一组 `\lambda_a`，让整体动作频率匹配 `t(a)`。

```python
def calibrate_multiaction(base_probs, prior_range, target, iters=80, lr=0.5):
    eps = 1e-8
    A = base_probs.shape[1]
    lam = np.zeros(A)

    for _ in range(iters):
        logits = np.log(np.clip(base_probs, eps, 1.0)) + lam[None, :]
        logits -= logits.max(axis=1, keepdims=True)
        q = np.exp(logits)
        q /= q.sum(axis=1, keepdims=True)

        marg = prior_range @ q
        grad = target - marg
        lam += lr * grad

        if np.max(np.abs(grad)) < 1e-4:
            break

    return q
```

这个方法的好处是：  
紧一点时，会先砍掉边界 combo；松一点时，会先加回边界 combo；不会破坏 solver 原始排序。

## 8. 范围更新怎么写

每观察到一个动作，就递推更新一次。

$$
R_{after}(h) \propto R_{before}(h)\cdot q(h,a^*)
$$

伪代码：

```python
def bayes_update(prior_range, action_probs_for_combo, observed_action_idx):
    post = prior_range * action_probs_for_combo[:, observed_action_idx]
    s = post.sum()
    if s <= 1e-12:
        return prior_range.copy()
    return post / s
```

整手牌开局到 hero 决策前，把所有已行动对手都更新一遍。

## 9. 非标准尺度怎么处理

尺度偏离分两部分：

## 9.1 价格效应

这个必须调。

例如标准 open 2bb，实际 3bb。  
先映射到最近 size 的 solver node，再按 size 弹性修正边界继续频率。

$$
continue_{target}
=
continue_{ref} + \eta_{arch}\cdot \log(size_{act}/size_{ref})
$$

如果 `size_act > size_ref`，通常 `continue_target` 会下降。

然后不是整体缩小，而是按 `score_call` / `score_defend` 从边界开始删。

---

## 9.2 范围信号效应

这个不能默认瞎调。

第一版建议：

- **默认只调价格，不调强弱**
- 只有当你有足够证据时，才把 size 当范围信号

证据可以来自：

- 该玩家在同位置的 size 选择非常稳定
- 或你有 showdown 数据证明“大 size 更强”

### 一个更好的做法

如果你的本地 solver 节点本来就有多个 size 动作，比如：

- `OPEN_2.0`
- `OPEN_2.5`
- `OPEN_3.0`

那观察到对手用了某个 size，本身就能更新范围：

$$
R(h|OPEN\_3.0) \propto R(h)\cdot p(h, OPEN\_3.0)
$$

如果实际是 2.7bb，没有 exact size，就对 size action 插值。

这比“3bb = 更强”这种粗暴规则好很多。

## 10. exploit 层怎么写

hero 也不要直接手工删牌，还是对 base policy 做偏移。

$$
\logit \pi_{hero}(h,a)
=
\logit \pi_{base}(h,a)
+
c \cdot \Delta(h,a)
$$

其中 `c` 是对手样本置信度，`Δ(h,a)` 来自 exploit 特征。

### 10.1 先预计算手牌特征

每手牌准备一些分数：

- `playability_score`
- `domination_risk`
- `blocker_score`
- `multiway_score`
- `linear_value_score`

这些第一版可以手工规则，后面再学。

---

### 10.2 常见 exploit 规则

#### 对紧 UTG open

- 降低 flat，特别是边缘 offsuit broadway
- 降低 bluff 3bet
- 保留最强 value 3bet
- 若对方 `fold_to_3bet` 很高，再给 blocker bluff 一点权重

示意：

```python
delta_call  = - conf * opener_strength_excess * domination_risk
delta_3bet  = - conf * opener_strength_excess * bluff_score
delta_3bet += + conf * overfold_to_3bet * blocker_score
```

#### 对松 UTG open

- 放宽 defend
- 放宽 value 3bet
- 若对方 open wide 且对 3bet 弃牌多，增加 blocker bluff 3bet

#### BTN 偷盲，SB/BB 防守不足

- 提高 open 频率
- 优先加入有 blocker、可实现率好的偷盲牌
- 少加纯垃圾 offsuit

```python
delta_open = conf * blind_underdefense * (
    0.5 * blocker_score + 0.5 * playability_score - 0.4 * domination_risk
)
```

#### 对手 cold call 过多

- squeeze 可以提高
- 但更偏 linear / value
- size 也应该更大

## 11. 你最关心的：怎么从“UTG open 10%”估计范围

最好的实现不是手写一堆牌，而是：

### 方法 A：有 solver 先验时

1. 取 `UTG unopened` 的 base open 概率 `p_base(h, OPEN)`
2. 用上面的 binary calibration，把整体 open 率压到 10%
3. 观察到他真的 open 了，再做一次 Bayes 更新：

$$
R(h|open) \propto R(h)\cdot q(h, open)
$$

这就是你程序里最该用的方法。

---

### 方法 B：没有 exact node 时

用离线预计算的 `score_open[UTG, unopened]` 构造 10% 模板范围：

```python
utg_open_10 = build_template_range(score_open_utg, 0.10)
```

这个模板自然比“全局 top10%”更像 solver。

---

### 为什么 call 不能按 top-x%

因为 call 不是线性顶端范围。

正确做法是：

```python
mp_call_7 = build_template_range(score_call_mp_vs_utg, 0.07)
```

这样构造出的 call range 会天然是中等牌力密集区，而不是把 `AA/KK/AK` 塞满。

## 12. 推荐的上下文分层回退

玩家 stats 很容易小样本，所以每个查询都做 backoff。

例如你要查：

`BTN facing CO open unopened, 100bb, IP`

按 4 层取：

1. `exact`: BTN vs CO open, 100bb
2. `mid`: BTN vs LP open
3. `coarse`: BTN vs open
4. `global`: 所有玩家 BTN facing open

然后按样本量加权：

$$
\hat p = \frac{\sum_l c_l \hat p_l}{\sum_l c_l},
\quad
c_l = \frac{N_l}{N_l + K_l}
$$

这样小样本不会乱跳。

```python
def get_hero_decision(hero_hand, history, player_profiles, stacks):
    state = build_runtime_state(history, stacks)

    # 1) 基准策略
    base_policy = node_resolver.resolve(state)

    # 2) 更新所有已行动对手范围
    opp_ranges = {}
    for player in acted_players_before_hero(history):
        r = initial_range()
        for event, ctx in player_action_sequence(player, history):
            target = player_model.get_target_mix(player, ctx, base_policy)
            combo_action_probs = calibrate_policy_for_player(
                base_policy_for_ctx(ctx),
                prior_range=r,
                target_mix=target,
                observed_size=event.size_bb
            )
            a_idx = action_index(combo_action_probs.actions, event)
            r = bayes_update(r, combo_action_probs.matrix, a_idx)
        opp_ranges[player.player_id] = r

    # 3) 调整 hero 策略
    hero_policy = exploit_adjuster.adjust(
        base_policy=base_policy,
        state=state,
        hero_hand=hero_hand,
        opp_ranges=opp_ranges,
        player_profiles=player_profiles
    )

    return hero_policy.action_probs_for(hero_hand)
```

## Step 6.4 实现平滑与 backoff 聚合

### 文件
`players/smoothing.py`, `players/backoff.py`

### 平滑公式
对每个动作频率：

$$
\hat p = \frac{n_a + k \mu}{N + k}
$$

其中：
- `n_a`：玩家该动作计数
- `N`：该上下文总机会数
- `\mu`：上一级 backoff 或 population prior
- `k`：先验强度

### 置信度公式

$$
c = \frac{N}{N + K}
$$

### 聚合方式
从 exact 到 coarse 逐层回退，逐层融合。

### 输出


## Step 9.2 knn 最近邻匹配

### 逻辑
exact miss 后，在同 family、同 actor_pos、同 stack bucket 的候选集合中找最近邻。

### 距离函数至少包含

- family mismatch
- actor_pos 距离
- aggressor_pos 距离
- limper_count 差异
- caller_count 差异
- players_left 差异
- ip 差异
- `log(size_ratio)` 差异
- `log(risk_over_pot ratio)` 差异

### 距离函数实现要求
使用显式权重，不允许黑盒模型。

### knn 合成
对前 k 个邻居：

$$
w_i = e^{-d_i/\tau}
$$

合成策略：

$$
P(h,a)=\frac{\sum_i w_i P_i(h,a)}{\sum_i w_i}
$$

### 输出
返回邻居 node_id、距离、权重。

### 验收
非标准 line 时能返回合理近邻，而不是空结果。

# 第十阶段：动作概率校准器

## Step 10.1 实现二元动作校准

### 文件
`players/player_model.py` 或 `ranges/range_engine.py`

### 适用场景
`OPEN/FOLD`、`LIMP/FOLD` 等二元动作。

### 公式
对基准概率 `p(h)`：

$$
q(h)=\sigma(\logit(p(h))+\lambda)
$$

通过求 `\lambda`，使整体动作频率匹配玩家目标频率。

### 约束
目标频率来自：

$$
t(a)=(1-c)\mu_{solver}(a)+c\hat p_{player}(a)
$$

### 验收
给定 `target_open=10%` 时，总体边际 open 频率误差 < 0.1%

---

## Step 10.2 实现多动作校准

### 适用场景
`FOLD/CALL/3BET`、`FOLD/CALL/4BET` 等。

### 公式

$$
q(h,a)=\frac{\exp(\log p(h,a)+\lambda_a)}{\sum_b \exp(\log p(h,b)+\lambda_b)}
$$

### 任务
实现一个数值求解器，输入：
- base_probs `[n_hands, n_actions]`
- prior_range `[n_hands]`
- target_mix `[n_actions]`

输出：
- calibrated_probs `[n_hands, n_actions]`

### 验收
- 各动作边际频率逼近 target_mix
- 每手牌各动作概率和为 1
- 不出现 nan

---

## Step 10.3 支持 size 维度的动作校准

### 逻辑
如果玩家统计有 size 分布，且样本足够，则对带 size 的 raise 动作做进一步拆分。

例：
- `OPEN_2.0`
- `OPEN_2.5`
- `OPEN_3.0`

### 要求
- 没样本时只调整 action family，不强行调整具体 size
- 样本足够时，先校准 action family，再校准 size mix

### 验收
同一玩家在 BTN 有固定 2.5bb open 习惯时，输出的 size mix 应向 2.5bb 倾斜。

# 第十二阶段：非标准 size 处理

## Step 12.1 size 插值

### 文件
`solver/node_resolver.py`

### 逻辑
若 solver 同 archetype 下有多个 size 节点，实际 size 在两者之间时，做插值。

### 推荐
使用 `log(size)` 插值：

$$
\alpha = \frac{\log s_2 - \log s}{\log s_2 - \log s_1}
$$

$$
P(s)=\alpha P(s_1)+(1-\alpha)P(s_2)
$$

### 验收
2.25bb 能在 2.0 和 2.5 节点之间平滑过渡。

---

## Step 12.2 size 外推 + 价格修正

### 逻辑
若实际 size 超出 solver 范围，例如库中只有 2bb，实际 3bb：

1. 先取最近 solver size 的基准策略
2. 用 `size_elasticity` 修正 continue 边际
3. 按动作专属 score 从边界删减/补回组合

### 要求
- 边界先动，不可整体平移
- defend/call 的修正优先影响边缘 offsuit 和 domination risk 高的牌
- pocket / suited connectivity / premium 变化较慢

### 验收
实际面对 3bb open 时，防守范围应比 2bb 时更紧。

---

## Step 12.3 仅在足够证据时启用 size-signal

### 逻辑
size 是否代表更强/更弱范围，只能在样本足够时启用。

### 启用条件
至少满足：
- `N >= SIZE_SIGNAL_MIN_SAMPLE`
- size bucket 与 baseline 的分布差异显著
- 或 showdown 反推支持这一结论

### 验收
默认情况下：
- 大小只影响价格，不影响强弱
- 只有高置信度玩家模型才启用 size=range signal

# chatgpt response

你的思路是对的，但我会把它从“**GTO 先验 + 玩家后验**”再往前补一层，变成：

**GTO 结构先验 + 池子先验 + 玩家后验。**

原因很简单：GTO 更适合当一个数学上自洽的 baseline，而不是直接执行的“标准答案”；真实牌局里，尤其小中级别，玩家会系统性地 **过度弃牌、过宽跟注、低频 bluff**，所以现代工具的工作流本身就是“拿你的实际 preflop 频率去对照 GTO benchmark，再按位置/对位/对 3bet 等场景找偏差”。GTO Wizard 的 GTO Reports 就是这么做的，而且还会给低样本提示；Jonathan Little 最近也把 GTO preflop 明确表述成 baseline，而不是机械执行的规则书。([GTO Wizard](https://blog.gtowizard.com/major-upgrade-gto-reports-have-arrived-plus-tons-of-new-features/))
先说结论：如果你在做程序，我最推荐的是下面这个组合。

1. **状态抽象 / 最近邻节点** 解决行动线偏差。  
2. **分层贝叶斯或 Dirichlet 平滑** 解决玩家个性偏差。  
3. **价格效应优先、强弱信号次之** 解决 size 偏差。  
4. **行动专属 hand ranking** 解决“open 10% / cold-call 8% 到底是哪几手牌”的问题。  
5. **高价值奇怪节点再做 nodelock / re-solve**，而不是所有 spot 都实时重算。  

这基本就是现在 solver/HUD/数据库分析三类工具合起来的常见工程做法。([GTO Wizard](https://blog.gtowizard.com/introducing-nodelocking/))
---

## 一、先别按“完整行动线字符串”建模，先按“公共状态”建模

如果你把 key 设计成 `UTG FOLD-MP FOLD-CO FOLD-BTN OPEN` 这种完整字符串，那一遇到 limp、cold call、5x open、非标准 3bet size 就会掉出树外。更稳的做法是把状态定义成“**公共信息状态**”：

- 当前行动者位置
- 面对的行动家族：`UNOPENED / VS_LIMP / VS_OPEN / VS_OPEN_CALL / VS_3BET / VS_4BET`
- aggressor 位置
- limper 数、caller 数
- 是否 IP/OOP
- 有效筹码
- `to_call`
- `raise_to_bb`
- `risk_over_pot`

这和扑克 AI 里常见的 action abstraction 思路是一致的：状态用 public information 表示，动作用抽象后的特征向量表示；现在的新一代多方 preflop solver 也在强调，真实牌局不是静态 chart，而是会出现 limper、奇怪 size、深浅筹码和不对称结构，所以需要“可映射”的状态表示而不是死 chart。([arXiv](https://arxiv.org/html/2403.04344v1))

## 二、几个常见方案

### 方案 A：规则层 exploit overlay
这是最常见、最好落地的第一版。

做法是：  
先拿最近的 GTO 节点当 baseline，再按玩家 stats 加几条明确规则：

- **UTG 比 GTO 更紧**：你对他 open 的 flat/call 收紧，bluff 3bet 减少，保留更纯的 value 3bet。  
- **BTN 偷盲时 SB/BB 防守不足**：BTN open 更宽。GTO Wizard 也明确把“盲位过于被动”对应到“更宽开池”当作 exploit 方向。([GTO Wizard](https://blog.gtowizard.com/punish_the_unstudied_preflop_mistakes_and_sizing_tells/))- **对手 fold to 3bet 高**：3bet bluff 扩宽；fold to 3bet 低：3bet 更偏强牌。Hand2Note 的 HUD 教学也是这个方向。([Hand2Note](https://hand2note.com/zh-cn/Blog/Features/key-preflop-stats-player-profiling-and-basic-adjustments))- **大 size open**：先别急着说对方更强，先按 pot odds 收紧继续范围。GTO Wizard 关于大 open 的文章明确写到：即便你认为对手没按更大 size 收紧开牌范围，你也仍然应该对更大的 raise 更常弃牌，因为价格变差了。([GTO Wizard](https://blog.gtowizard.com/how-to-respond-to-large-preflop-raises-in-poker/))
这个方案优点是快、可解释、跟现有 HUD 逻辑兼容。缺点是容易变成很多 if/else，尤其 limp 和 cold call 多起来之后会越来越乱。

---

### 方案 B：分层贝叶斯 / Dirichlet 后验更新
这是我最推荐的“主干方案”。

学术上，扑克对手建模早就有 “**对对手策略分布建先验，然后根据观察更新 posterior**” 的做法；相关工作甚至明确比较过独立 Dirichlet prior 和 informed prior。([arXiv](https://arxiv.org/pdf/1207.1411))
你这里可以这么做。对每个上下文 `c` 和动作集合 `A(c)`：

$$
\alpha_0(c,a)=\tau_{gto} \cdot p_{gto}(a|c) + \tau_{pool} \cdot p_{pool}(a|c)
$$

观察到玩家在该上下文的动作计数后：

$$
\alpha(c,a)=\alpha_0(c,a)+n(c,a)
$$

后验均值：

$$
\mathbb E[p(a|c,\text{player})]=\frac{\alpha(c,a)}{\sum_{a'}\alpha(c,a')}
$$

这里的关键不是公式本身，而是这两个细节：

第一，**不要纯 GTO prior**。  
因为真实池子本身就系统性偏离 GTO；你用 “GTO 结构 + pool 频率 + player 数据” 比只用 GTO 稳得多。

第二，**要做分层 backoff**。  
比如 exact context 样本不足时，退到：
- exact size → size bucket
- exact aggressor pos → aggressor group（EP/MP/LP/Blind）
- `BTN vs CO open` → `BTN vs LP open`
- `BTN vs open` → `BTN global`

这个方案的优点是：  
你可以自然地把“小样本玩家”和“已知 regular”统一到同一个框架里，而且 exploit 强度会随样本自动变化，不容易过拟合。

---

### 方案 C：玩家 archetype / 聚类模型
这是很多 HUD 或分析系统很常用但不总被明说的一层。

你可以先不直接记“这个玩家每个点位都是什么频率”，而是先把玩家压成一个 archetype mixture，比如：

- nit / TAG / LAG / whale / limp-passive / loose caller / overfolder / size-tell 型

聚类时用的特征，不要只用 VPIP/PFR/3bet 这三项，而要用：

- VPIP
- PFR
- VPIP-PFR gap
- RFI by position
- 3bet by position
- fold to 3bet IP/OOP
- cold call by position
- blind defend vs steal
- limp / limp-fold / limp-reraise
- size bucket 分布
- size entropy（size 是否很固定）

Hand2Note 的文章也明确建议把 preflop 画像拆到位置、IP/OOP、3bet、fold to 3bet，而不是只看总表。([Hand2Note](https://hand2note.com/zh-cn/Blog/Features/key-preflop-stats-player-profiling-and-basic-adjustments))
这个方案适合两类场景：

- 样本还不够，先用 archetype 填补空白
- live pool / 小样本匿名池，需要快速判断是“宽松跟注站”还是“紧被动常客”

缺点是实现稍复杂，而且 archetype 的边界会漂。

---

### 方案 D：精确 nodelock / re-solve
这是精度最高、也最贵的方案。

现在的 solver 工作流已经很明确了：  
当你知道对手偏差时，最“业内标准”的做法不是拍脑袋，而是 **nodelock 对手策略，然后看 solver 的最优 counterstrategy**。GTO Wizard 的 nodelocking 就支持按 combo、hand class 或 action frequency 去锁定策略，并让 solver 自动给出 exploit 响应；它的新多方 preflop solver 也已经支持 custom sizes、limps、cold-call sizes、玩家 profile 和 nodelocking。([GTO Wizard](https://blog.gtowizard.com/introducing-nodelocking/))
这个方案适合：

- 你要离线生成一批“非常重要的非标准节点”
- 某个 live pool 明显大量 5x open / limp / cold-call
- 你愿意花计算预算做高价值子树

不适合：
- 全量实时在线决策
- 你的历史数据还很脏、玩家模型还不稳时

## 三、三种偏差分别怎么处理

### 1）玩家个性偏差
这是最适合 posterior 的部分。

你最终要学的是：  
**在某个 context 下，这个玩家偏离 GTO 的动作频率是多少。**

最有用的 stats 不是笼统 VPIP/PFR，而是节点化的：

- `RFI by position`
- `open limp`
- `iso vs limpers`
- `cold call vs open`
- `3bet vs open`
- `squeeze vs open+call`
- `fold/call/4bet vs 3bet`
- `fold/call/jam vs 4bet`
- `SB/BB defend vs BTN/CO steal`
- `limp-fold / limp-call / limp-reraise`
- `fold to squeeze as cold caller`

GTO Reports 现在的 preflop 对比项本身就覆盖了 VPIP、PFR、RFI、Limp、Squeeze、3bet、4bet、5bet，并支持位置拆分和 `vs 3bet` 防守拆分；这是个很好的 stats 设计蓝图。([GTO Wizard](https://blog.gtowizard.com/major-upgrade-gto-reports-have-arrived-plus-tons-of-new-features/))
---

### 2）size 偏差
这里一定要拆成两部分：

**价格效应** 和 **信息效应**。

- **价格效应**：无论如何都应该用。大 raise 让你拿更差的价格，所以继续范围该收紧。这个不需要知道对手是不是“更强”。([GTO Wizard](https://blog.gtowizard.com/how-to-respond-to-large-preflop-raises-in-poker/))- **信息效应**：只有当玩家长期表现出“size 跟牌力相关”时才用。GTO Wizard 也专门提到，不懂理论的玩家常常会按手牌类别选 size，也就是所谓 sizing tells。([GTO Wizard](https://blog.gtowizard.com/punish_the_unstudied_preflop_mistakes_and_sizing_tells/))
所以默认规则应该是：

- **先按价格修正**
- **只有在样本足够 + showdown 或稳定 size 分布支持时，才把 size 当范围强弱信号**

这会避免你犯一个很常见的错：把所有 3x、4x 都自动解释成“更强范围”。

---

### 3）行动线偏差：多了 limp、cold call、weird line
这里最忌讳的是把它们硬塞回标准树里。

更好的做法是：

- limp 作为单独 action family
- open+cold call 作为单独 family
- 对 weird line 先映射到最近公共状态，再取最近邻 solver 节点或模板

冷跟范围尤其不要按“top x% 手牌”来理解。GTO Wizard 最近关于 range morphology 的文章把 range 分成 linear / polarized / condensed / merged，其中 **condensed range 是以中等牌力为主**。这正好解释了为什么很多 preflop call/cold-call 不是线性的 top-x：顶端强牌不少已经 3bet 了，底端弱牌已经 fold 了，剩下往往是中间密集区。([GTO Wizard](https://blog.gtowizard.com/preflop-range-morphology/))
所以如果一个玩家显示“MP 对 UTG 冷跟 8%”，你的程序不应该直接取“最强 8% 手牌”，而应该取 **CALL-specific 排序靠前的 8%**。

---

## 四、玩家历史数据怎么存：不要只存 stats，要存“机会”

这是最关键的一点。

HUD 里这些 preflop stats 的定义，本质上全是：

**某动作发生次数 / 该动作有机会发生的次数**

比如 DriveHUD 的定义里：

- `BB Call BTN Steal% = 面对 BTN steal 时 BB 跟注次数 / 面对 BTN steal 且可跟注的次数`
- `Fold to 3Bet% = 面对 3bet 后弃牌次数 / 面对 3bet 且可弃牌的次数`
- `Limp Fold% = limp 后弃牌次数 / limp 后面临加注且可弃牌的次数`
- `Fold to Squeeze as Cold Caller% = 冷跟后面对 squeeze 弃牌次数 / 冷跟后面对 squeeze 且可弃牌的次数`。([Poker HUD | Stats | Tracking](https://drivehud.com/dh2-knowledge-base/what-are-the-hud-stat-definitions/))
所以数据层我建议至少分三层。

### 第一层：原始 decision-opportunity 日志
每一行不是“一手牌”，而是“一个玩家在一个上下文里的一个决策机会”。

建议字段：

```text
hand_id
timestamp
player_id
table_format          # 6max/9max, cash/MTT
stakes
rake_profile
ante_flag

actor_pos
aggressor_pos
family                # UNOPENED / VS_LIMP / VS_OPEN / VS_OPEN_CALL / VS_3BET / VS_4BET
limper_count
caller_count
players_left
ip_oop

effective_bb
to_call_bb
pot_before_bb
raise_to_bb
risk_over_pot         # 额外投入 / 当前底池
size_bucket
stack_bucket

legal_actions         # e.g. [FOLD, CALL, 3BET]
chosen_action_family  # e.g. CALL
chosen_size_bb
chosen_size_bucket

is_steal_spot
is_facing_steal
is_facing_3bet
is_facing_squeeze
is_after_limp
is_after_cold_call

showdown_revealed     # optional
revealed_combo        # optional, e.g. AsQs
revealed_class169     # optional, e.g. AQs
```

这层是最重要的，因为后面所有统计、回放、修正、再训练都要靠它。

---

### 第二层：聚合 stats cube
按 `context_key` 做聚合，统计：

```text
player_id
context_key

opportunity_n
action_count[FOLD]
action_count[LIMP]
action_count[CALL]
action_count[OPEN]
action_count[ISO]
action_count[3BET]
action_count[SQUEEZE]
action_count[4BET]
action_count[JAM]

size_count[action_family][size_bucket]

last_seen
decayed_opportunity_n
decayed_action_counts
```

这里的 `context_key` 我建议是：

```text
(format, rake_bucket, ante_flag,
 actor_pos, family, aggressor_pos_group,
 limper_count_bucket, caller_count_bucket,
 ip_oop, effective_stack_bucket, size_bucket)
```

然后再做 4 到 5 层 backoff。

---

### 第三层：showdown evidence
这是很多人会漏掉的，但它对“size tell”和“冷跟范围形状”特别有用。

你单独存：

```text
player_id
context_key
action_family
size_bucket
revealed_combo / revealed_class169
count
```

它的作用不是主导频率，而是辅助判断：

- 这个人 4x open 是否真的更 value-heavy
- limp-reraise 里 trap 比例多不多
- cold-call 是否特别偏 pocket / suited broadway
- 某些固定 size 是否几乎只拿强牌

因为频率 stats 覆盖高但看不到具体牌，showdown 反过来是覆盖低但信息密。

## 五、我建议你直接建模的参数

如果只能挑一批最值钱的参数，我会优先这些。

### 用来做玩家画像的
- VPIP
- PFR
- VPIP-PFR gap
- RFI by position
- 3bet by position
- fold to 3bet IP/OOP
- cold call by position
- squeeze
- fold to squeeze
- defend vs steal by blind/steal-position
- limp / limp-fold / limp-call / limp-reraise

- sample_n / decayed_sample_n

这些都能从现成 HUD 体系里找到对应思路；Hand2Note 也明确强调样本量很重要，rough heuristic 上大约要上千手才开始较可靠地读 3bet / fold to 3bet 这类 stats，而 GTO Reports 也会对低样本区域专门做置信度提示。([Hand2Note](https://hand2note.com/zh-cn/Blog/Features/key-preflop-stats-player-profiling-and-basic-adjustments))
### 用来做 range 更新的
- `P_GTO(action | hand, context)`
- `P_player(action | context)` 的 posterior

- showdown 修正项

最终更新可以写成：

$$
P(h \mid obs, player, c)\propto P_{prior}(h\mid c)\cdot P_{player}(a,s\mid h,c)
$$

其中：

- `P_prior(h|c)` 建议来自 GTO 节点或最近邻节点  
- `P_player(a,s|h,c)` 则是把 GTO 的 combo-level action matrix 按玩家 posterior 做频率平移之后得到的

## 六、如果你只有一版资源，最推荐怎么做

我会建议你这样落地。

### 第一步
做一个 **state resolver**：

- exact GTO 节点就直接用
- 没有就按公共状态找最近邻
- limp / open+call / weird size 用模板回退

### 第二步
对每个 `context_key` 建一个 **Dirichlet/贝叶斯 stats 模型**：

```text
prior = tau_gto * GTO + tau_pool * population
posterior = prior + observed_counts
```

### 第三步
对 raise 动作再单独建一个 **size posterior**：

```text
P(size_bucket | OPEN, context, player)
P(size_bucket | 3BET, context, player)
...
```

### 第四步
为每个 context 预计算 **行动专属 hand score**：

```text
score_open[h|c]  = P_GTO(OPEN | h,c)
score_call[h|c]  = P_GTO(CALL | h,c)
score_3bet[h|c]  = P_GTO(3BET | h,c)
```

这样：
- “UTG open 10%” 用 `score_open`
- “MP cold-call 8%” 用 `score_call`

而不是都用同一个全局牌力排序。

### 第五步
默认规则上：

- size 先只影响价格，不影响强弱
- 只有高样本 + showdown 证据时，才启用 size-signal

### 第六步
高价值怪节点再去做 nodelock / custom solve

这样你会得到一个很稳的系统：  
**平时靠 resolver + posterior 快速工作，难点位靠 nodelock 离线补库。**

## 七、我对你这个项目最重要的一句建议

不要把“玩家历史数据”建成一堆静态 HUD 百分比。  
要把它建成：

**按 context 统计的 action posterior + size posterior + showdown evidence。**

这样你才能同时回答这三件事：

1. 这个人 **在这个位置/节点** 会不会偏离 GTO  
2. 他偏离的是 **频率** 还是 **size 选择**  
3. 这个偏离会把他的范围变成 **更线性、更极化，还是更 condensed**

其中 GTO 给你“结构”，pool 给你“现实池子平均”，archetype 给你“相似玩家画像”，parent-context 给你“小样本借力”。这种“先验 + 观察 + 后验”的 poker 对手建模本身就是学术和工程里都很自然的路线；Bayes’ Bluff 这类工作就是用 prior 和观察动作去推 posterior，对简化游戏用过 Dirichlet prior，对完整德州也强调过更 informed 的 prior。([arXiv](https://arxiv.org/abs/1207.1411))

## 一、几种常见且可落地的方案

### 方案 1：Empirical Bayes 平滑
这是最稳、最好上线的第一版。

对二元动作，比如 `BTN unopened 是否 open`，用 Beta-Binomial；  
对多动作节点，比如 `BB vs BTN open: fold / call / 3bet`，用 Dirichlet-Multinomial。

二元动作可直接写成：

$$
p_{\text{post}}=\frac{k_{gto}p_{gto}+k_{pool}p_{pool}+k_{arch}p_{arch}+x}{k_{gto}+k_{pool}+k_{arch}+n}
$$

- `x` 是该动作的实际次数  
- `n` 是该动作有机会发生的次数  
- `k_*` 是各类先验强度

多动作用：

$$
\alpha_a = k_{gto}p_{gto}(a)+k_{pool}p_{pool}(a)+k_{arch}p_{arch}(a)+n_a
$$

$$
p_{\text{post}}(a)=\frac{\alpha_a}{\sum_b \alpha_b}
$$

这个方案的优点是实现简单、可解释、数据库友好。绝大多数 HUD/统计产品本质上都在围绕“机会数 + 动作数”做频率估计，只是很多工具没把它显式写成 Bayesian 形式。([Poker HUD | Stats | Tracking](https://drivehud.com/dh2-knowledge-base/what-are-the-hud-stat-definitions/))
适合你第一期就上。

---

### 方案 2：分层 backoff / partial pooling
这是我最推荐和方案 1 组合使用的做法。

不要只看 exact context。你应该给每个 context 定义一条回退链，例如：

1. `BTN vs CO open 2.5bb 100bb`  
2. `BTN vs LP open 2.5bb 100bb`  
3. `BTN vs LP open 100bb`  
4. `BTN vs open 100bb`  
5. `BTN vs open`  
6. `pool global`

然后 exact 样本不足时，把更粗粒度 posterior 当成更细粒度 prior：

$$
\alpha^{(exact)}_a
=
n^{(exact)}_a
+\lambda_1 p^{(parent1)}_a
+\lambda_2 p^{(parent2)}_a
+\lambda_3 p^{(pool)}_a
+\lambda_4 p^{(gto)}_a
$$

这样做的好处是：  
你既不会因为 exact 样本太少而乱跳，也不会把所有信息都粗暴压回总池子平均。

这和当前工具越来越强调“位置对位置”“面对某个动作”的拆分方向是一致的：GTO Reports 本身就把 preflop 分到位置、position-vs-position 和 facing 3-bet 这类上下文，Hand2Note 也明确说 overall 3-bet stat 不够，要看具体位置和动态 HUD。([GTO Wizard](https://blog.gtowizard.com/major-upgrade-gto-reports-have-arrived-plus-tons-of-new-features/))
---

### 方案 3：相似玩家 / archetype 先验
这是小样本最有用的一层。

Hand2Note 的官方文档直接说了：某些 spot 样本不足时，可以选一组相似玩家，把他们聚合成一个统计报告来估计范围。现在的 solver 工作流也越来越支持 player profiles；GTO Wizard 甚至已经把自定义对手 profile、limping、custom raise sizes、cold-call sizes、nodelocking 放进多方 preflop 求解里。([Hand2Note](https://hand2note.com/Help/Features/statistics))
你落地时可以先做最简单的 archetype：

- nit / 紧弱 regular
- TAG
- LAG
- loose-passive caller
- limp-passive
- overfolder
- aggro 3bettor

然后为每个 archetype 维护 `context -> action posterior`。  
未知玩家一开始先继承 `pool + archetype`，随着个人样本增加，再慢慢让个人数据接管。

这是实际工程里非常值的一层，因为很多玩家你永远拿不到足够 exact 样本，但你可以很快知道他更像哪一类人。

---

### 方案 4：长周期 + 近期窗口双轨
这不是最“学术正统”，但非常实用。

给每个玩家同时维护：

- `lifetime_counts`
- `recent_counts`（例如指数衰减或近 2k 手）

然后：

$$
p = w_{recent}p_{recent} + (1-w_{recent})p_{lifetime}
$$

这么做是为了处理两种现实情况：

- 玩家风格真的在变
- 你观察到的是短期 tilt / session-specific 偏差

工程上我建议 raw integer counts 和 decayed float counts 都存。raw 用来审计，decayed 用来在线查询。

---

### 方案 5：高价值节点做 nodelock / profile library
这不是每个 spot 都用，而是拿来处理“高频且高价值的怪节点”。

比如：

- 你这个池子里大量 4x open
- 某些桌普遍 limp/cold call
- 某类玩家 SB defend 严重不足
- 某类玩家 BTN 固定 2.5x open、BB 固定大 3bet

这种场景，不要只靠简单频率平移。更常见的业内做法是：先根据 player/profile 锁定策略，再让 solver 给 counterstrategy。GTO Wizard 的 nodelocking 和 custom profiles 本质就是这个工作流。([GTO Wizard Help](https://help.gtowizard.com/how-to-use-nodelocking/))
我的建议是把它当成“离线补库层”，不是在线主链路。

---

## 二、我最推荐的生产级组合

如果你要真的落地，我推荐：

**主链路：方案 1 + 方案 2 + 方案 3**  
**增强层：方案 4**  
**离线校准层：方案 5**

具体公式我会写成：

$$
\alpha_a
=
k_{gto}p_{gto}(a|c)
+
k_{pool}p_{pool}(a|c,s)
+
k_{arch}p_{arch}(a|c)
+
k_{parent}p_{parent}(a|c)
+
n_{player}(a|c)
$$

其中：

- `c` = context  
- `s` = pool segment  
- `n_player(a|c)` = 这个玩家在这个 context 的动作计数  
- `p_pool` 最好按站点/级别/人数/ante/rake/stack 分层  
- `p_gto` 用你的 solver baseline

然后：

$$
p_{post}(a|player,c)=\frac{\alpha_a}{\sum_b \alpha_b}
$$

我会这样设默认逻辑：

- 样本很少：主要信 `pool + archetype + gto`
- 样本一般：开始让 `player exact context` 进入
- 样本足够：player posterior 主导
- 极怪 size / limp / cold call 线：走 profile / nodelock 离线库

---

## 三、数据库别存“百分比”，要存“充分统计量”

这是最重要的设计点。

不要只存：

- VPIP = 27
- 3bet = 8
- fold_to_3bet = 61

因为这样你丢掉了最重要的信息：**样本量和不确定性**。

你应该至少存三层数据：

### 1）原始决策机会日志
一行代表“这个玩家在这个 context 下有一次决策机会”。

### 2）聚合计数
按 `player_id + context_id` 聚合出 opportunity 和各动作计数。

### 3）先验表
按 `pool_segment + context_id` 和 `archetype + context_id` 存先验计数或先验概率。

这样你才能随时重算 posterior，也能替换 smoothing 逻辑，而不是被死百分比绑死。

## 四、推荐的数据结构

```sql
create table pool_segment_dim (
```

## 五、在线查询怎么做

给定 `player_id + context_id`，我会按这个顺序查：

1. exact `player_context_action_stats`
2. parent/backoff context
3. `archetype_context_action_prior`
4. `pool_context_action_prior`
5. `gto baseline`

然后：

```text
alpha = k_gto * gto_probs
      + k_pool * pool_probs
      + k_arch * archetype_probs
      + k_parent * parent_probs
      + player_exact_counts
```

最后：

```text
posterior_probs = alpha / sum(alpha)
confidence = opp_n / (opp_n + k_total)
```

对 size：

```text
beta = l_pool * pool_size_probs
     + l_arch * arch_size_probs
     + l_parent * parent_size_probs
     + player_size_counts

posterior_size = beta / sum(beta)
```

这一套很适合数据库，因为你真正要查的就是几张聚合表，不是全扫原始手牌。

## 七、我给你的最终建议

如果你现在就要开工，我会这样定：

**数据库层只存“机会 + 计数 + 先验来源”，不存死百分比。**  
**模型层用 Dirichlet/Beta 做平滑。**  
**查询层按 exact → backoff → archetype → pool → gto 的顺序借力。**  
**范围层再把这个 posterior action mix 回灌到 GTO combo 矩阵。**

这样你的小样本问题就不会变成“这人 6 次机会里 3bet 了 2 次，所以他是 33% 3better”这种灾难；而是会变成“这人目前略偏激进，但仍强烈收缩到 pool/archetype/GTO 先验”的可控后验。Hand2Note 的相似玩家聚合、GTO Reports 的低样本置信度与位置矩阵、以及 GTO Wizard 的 custom profiles / nodelocking，本质上都在说明同一件事：真实对手建模不是看一个裸百分比，而是看带上下文、带置信度、带先验的策略分布。([Hand2Note](https://hand2note.com/Help/Features/statistics))
下一步最合适的是把这几张表直接写成一版 PostgreSQL DDL，并把 `posterior_action_mix(player_id, context_id)` 这个查询函数也一起设计出来。

Dirichlet/Beta 做平滑 详细解释如何做，比如整个玩家池 utg 第一次行动这个spot 有200万手数据 其中有2万次raise了，然后我观察到一个玩家10次在utg raise了5次，如何根据先验来平滑？

# chatgpt response

可以。你这个例子最适合用 **Beta-Binomial** 来讲，因为这是一个二元动作：

- 在 `UTG 第一次行动` 这个 spot
- 动作简化成：
  - `RAISE`
  - `NOT_RAISE`

如果以后你要扩展成：

- `FOLD / LIMP / OPEN 2bb / OPEN 2.5bb / OPEN 3bb`

那就从 Beta 升级成 **Dirichlet-Multinomial**。  
但先把 Beta 彻底讲清楚最重要。

---

# 1. 你的例子先翻译成统计问题

你给的数据是：

- 整个玩家池在 `UTG 第一次行动` 这个 spot 有 **2,000,000 次机会**
- 其中 **20,000 次 raise**

所以池子平均 raise 率是：

$$
p_{pool} = \frac{20000}{2000000} = 0.01
$$

也就是 **1%**

然后某个玩家：

- 在这个 spot 只观察到 **10 次机会**
- 其中 **5 次 raise**

原始频率是：

$$
p_{raw} = \frac{5}{10} = 0.5
$$

也就是 **50%**

显然这个 50% 不可信，因为样本太少。  
你真正想做的是：

> 用“池子平均 1%”作为先验，再结合这个玩家的 10 次观察，得到一个更合理的后验 raise 概率。

---

# 2. Beta-Binomial 是什么

Beta 分布是二元概率的先验分布。  
如果你把“某玩家在这个 spot 的真实 raise 概率”记成 $\theta$，那么：

$$
\theta \sim Beta(\alpha,\beta)
$$

这里：

- $\alpha$ 可以理解成“先验里的 raise 伪计数”
- $\beta$ 可以理解成“先验里的 not-raise 伪计数”

如果再观察到玩家：

- raise 了 $x$ 次
- 总共 $n$ 次机会

那么后验就是：

$$
\theta \mid data \sim Beta(\alpha + x,\; \beta + n - x)
$$

后验均值是：

$$
E[\theta \mid data] = \frac{\alpha + x}{\alpha + \beta + n}
$$

这就是最常用的平滑公式。

---

# 3. 先验怎么从玩家池数据得到

池子平均 raise 率是 1%，但你不能直接把 2,000,000 和 20,000 原封不动塞进 Beta 先验：

$$
\alpha = 20000,\quad \beta = 1980000
$$

因为这样先验强度太大了。  
这个玩家就算打了几百手，也几乎动不了先验。

所以要做 **缩放先验强度**。

定义一个超参数 $k$，表示“你愿意让池子先验相当于多少个伪样本”。

然后设：

$$
\alpha = k \cdot p_{pool}
$$

$$
\beta = k \cdot (1-p_{pool})
$$

如果池子平均是 1%，那就是：

$$
\alpha = 0.01k,\quad \beta = 0.99k
$$

---

# 4. 用你的例子实际算

你的玩家数据：

- $x = 5$
- $n = 10$

池子平均：

- $p_{pool}=0.01$

---

## 情况 A：先验强度 $k=20$

那么：

$$
\alpha = 20 \cdot 0.01 = 0.2
$$

$$
\beta = 20 \cdot 0.99 = 19.8
$$

后验：

$$
E[\theta \mid data]
=
\frac{0.2 + 5}{0.2 + 19.8 + 10}
=
\frac{5.2}{30}
\approx 0.1733
$$

也就是 **17.3%**

原始是 50%，平滑后变成 17.3%。

---

## 情况 B：先验强度 $k=100$

$$
\alpha = 1,\quad \beta = 99
$$

后验：

$$
E[\theta \mid data]
=
\frac{1+5}{1+99+10}
=
\frac{6}{110}
\approx 0.0545
$$

也就是 **5.45%**

---

## 情况 C：先验强度 $k=500$

$$
\alpha = 5,\quad \beta = 495
$$

后验：

$$
E[\theta \mid data]
=
\frac{5+5}{5+495+10}
=
\frac{10}{510}
\approx 0.0196
$$

也就是 **1.96%**

---

# 5. 这三个结果说明什么

同样的数据：

- 观察值：10 次里 raise 5 次
- 原始频率：50%

平滑后结果会因为先验强度不同而差很多：

- $k=20$ → 17.3%
- $k=100$ → 5.45%
- $k=500$ → 1.96%

所以真正的问题不是“会不会平滑”，而是：

> **先验强度 k 该设多少？**

---

# 6. 这个公式其实就是加权平均

后验均值可以改写成：

$$
E[\theta \mid data]
=
\frac{k \cdot p_{pool} + x}{k+n}
$$

再整理一下：

$$
E[\theta \mid data]
=
\frac{k}{k+n}p_{pool} + \frac{n}{k+n}\frac{x}{n}
$$

也就是：

$$
p_{post} = w_{prior} \cdot p_{pool} + w_{data} \cdot p_{raw}
$$

其中：

$$
w_{prior} = \frac{k}{k+n},\quad w_{data}=\frac{n}{k+n}
$$

这很直观：

- 样本少时，主要信先验
- 样本多时，主要信玩家自身数据

比如 $k=100, n=10$：

$$
w_{prior} = \frac{100}{110}=0.909
$$

$$
w_{data} = \frac{10}{110}=0.091
$$

所以：

$$
p_{post}=0.909\times 0.01 + 0.091\times 0.5
=0.0545
$$

这和上面一样。

---

# 7. 如何选 k

这才是落地关键。

## 方案 1：手工设定
最简单。

比如：

- 高频常见 spot：`k = 50 ~ 200`
- 极少见 spot：`k = 200 ~ 1000`

直觉上：

- 如果你觉得“10 次观察不该太可信”，就设大一些
- 如果你觉得“10 次观察已经有点信息”，就设小一些

### 一个常用解释
如果设：

$$
k=100
$$

就相当于说：

> 池子平均相当于“额外 100 次伪观察”

---

## 方案 2：按 context 稀有度设 k
更合理。

例如：

- `UTG unopened open`：机会很多，群体稳定，设大一点，比如 200
- `SB limp-reraise facing iso`：机会很少，行为噪声大，也许设 50 或者走更高层 backoff

注意这里不是“样本多就一定小 k”，而是看这个 spot 的 **群体离散程度**。

如果群体中不同玩家在这个 spot 差异很大，就不能设太大的 k。  
如果群体非常稳定，k 可以大。

---

## 方案 3：从群体方差反推 k
这是更“正统”的 Empirical Bayes。

Beta 分布的均值和方差是：

$$
E[\theta]=\mu = \frac{\alpha}{\alpha+\beta}
$$

$$
Var(\theta)=\frac{\mu(1-\mu)}{\alpha+\beta+1}
$$

如果写成：

$$
\alpha = k\mu,\quad \beta = k(1-\mu)
$$

则有：

$$
Var(\theta)=\frac{\mu(1-\mu)}{k+1}
$$

所以：

$$
k \approx \frac{\mu(1-\mu)}{Var(\theta)} - 1
$$

意思是：

- 如果群体里玩家都很像，方差很小，$k$ 很大
- 如果群体差异很大，方差很大，$k$ 很小

这比拍脑袋设 k 更合理。

### 落地方法
你在数据库里可以对 `UTG unopened` 这个 context，把“每个玩家的长期 raise 率”拿出来，估计横截面方差，再反推一个 k。

# 8. 如果你不只想用池子平均，还想用 GTO 先验怎么办

你之前说得对：  
你想用 **GTO 作为先验，再结合玩家池和个人样本做后验**。

那最简单的做法是把多个先验混合成一个“总先验”。

比如：

- $p_{gto}$：GTO 在这个 spot 的 raise 率
- $p_{pool}$：玩家池平均 raise 率
- $p_{arch}$：该玩家所属 archetype 的平均 raise 率

然后设：

$$
\alpha = k_{gto}p_{gto} + k_{pool}p_{pool} + k_{arch}p_{arch}
$$

$$
\beta = k_{gto}(1-p_{gto}) + k_{pool}(1-p_{pool}) + k_{arch}(1-p_{arch})
$$

再加上玩家数据：

$$
\theta \mid data \sim Beta(\alpha + x,\beta + n-x)
$$

后验均值：

$$
E[\theta|data]
=
\frac{\alpha + x}{\alpha+\beta+n}
$$

这样就不是单纯向池子平均回归，而是向“**GTO + pool + archetype 的混合先验**”回归。

---

## 举个数值例子

假设：

- $p_{gto}=12\%$
- $p_{pool}=1\%$
- $p_{arch}=4\%$

先验强度设：

- $k_{gto}=20$
- $k_{pool}=60$
- $k_{arch}=20$

那么：

$$
\alpha = 20 \cdot 0.12 + 60 \cdot 0.01 + 20 \cdot 0.04
= 2.4 + 0.6 + 0.8 = 3.8
$$

总强度：

$$
k_{total}=20+60+20=100
$$

所以：

$$
\beta = 100 - 3.8 = 96.2
$$

如果玩家观察还是 `10 次里 raise 5 次`：

$$
E[\theta|data]
=
\frac{3.8+5}{100+10}
=
\frac{8.8}{110}
= 8.0\%
$$

这个结果就比“只向池子 1% 回归”更合理，因为你保留了 GTO 结构和 archetype 信息。

# 9. 如果动作不止 raise / not raise，而是多动作怎么办

这时用 **Dirichlet**。

假设在 `UTG unopened` 这个 spot，你记录的是：

- `FOLD`
- `LIMP`
- `OPEN_SMALL`
- `OPEN_BIG`

那么先验：

$$
(\theta_1,\theta_2,\theta_3,\theta_4)\sim Dirichlet(\alpha_1,\alpha_2,\alpha_3,\alpha_4)
$$

观察到玩家数据后：

$$
\alpha_i^{post}=\alpha_i^{prior}+n_i
$$

后验均值：

$$
E[\theta_i|data]=\frac{\alpha_i}{\sum_j \alpha_j}
$$

### 举例
先验你可以用：

- pool 平均
- GTO action mix
- archetype 平均

来构造：

$$
\alpha_i
=
k_{gto}p_{gto,i}
+
k_{pool}p_{pool,i}
+
k_{arch}p_{arch,i}
$$

然后加玩家观察次数。

# 12. 这个方法的本质解释

你可以把 Beta/Dirichlet 平滑理解成：

> “我假装在真正观察到这个玩家之前，已经先看到了若干个虚拟样本，这些虚拟样本的分布来自 pool/GTO/archetype。然后再把这个玩家的真实观察加上去。”

这就是为什么它特别适合扑克：

- 小样本时，玩家数据不可靠 → 靠先验稳住
- 大样本时，玩家风格清晰 → 真实数据逐渐压过先验

# 一、Dirichlet 是什么（直观理解）

Dirichlet 就是 **多项概率的 Beta 分布扩展**。

Beta 用在：

```
raise / not raise
```

Dirichlet 用在：

```
fold / call / 3bet
```

Beta：

```
θ ~ Beta(α, β)
```

Dirichlet：

```
(θ1, θ2, θ3) ~ Dirichlet(α1, α2, α3)
```

含义：

```
α1 = fold 的伪计数
α2 = call 的伪计数
α3 = 3bet 的伪计数
```

总强度：

```
K = α1 + α2 + α3
```

期望值：

```
E[θi] = αi / K
```

# 九、如何用这个 posterior 调整 GTO range

现在你已经有：

```
posterior action mix
```

例如：

```
fold = 0.575
call = 0.358
3bet = 0.067
```

但 solver baseline 是：

```
fold = 0.55
call = 0.35
3bet = 0.10
```

我们要 **调整 GTO matrix**。

常用方法：

### logit shift

例如：

```
Δ_call = log(0.358 / 0.35)
Δ_3bet = log(0.067 / 0.10)
```

再对每个 combo：

```
P_new(a|h)
= softmax(log(P_gto(a|h)) + Δ_a)
```

这样：

- 边界 combo 会调整
- 强牌结构保持
- 不会破坏 solver 排序
